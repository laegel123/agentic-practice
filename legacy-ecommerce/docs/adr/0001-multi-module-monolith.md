# ADR-0001: 멀티모듈 모놀리스 구조

- **상태**: Accepted
- **날짜**: 2026-06-12 (사후 기록 — 결정 시점 불명)
- **결정자**: 원 개발팀 (복원)

## 맥락 (Context)

사내 커머스 백엔드를 상품/주문, 결제, 관리자, 배치 등 여러 책임으로 나눠야 했다.
팀 규모는 작고 운영 인프라(서비스 디스커버리, 메시지 브로커, 분산 트레이싱 등)는 없었다.

## 결정 (Decision)

하나의 Gradle 멀티모듈 저장소로 구성하되, 책임별로 **독립 실행 가능한 Spring Boot 앱**
(`ecommerce-service`, `payment-service`, `admin`, `batch`)으로 나눈다. 공통 코드는
`core-framework`(공통 기반 클래스)와 `common-util`(정적 유틸) 두 라이브러리 모듈로 추출한다.
앱들은 같은 저장소·같은 빌드에서 함께 관리되지만 별도 프로세스로 뜬다.

## 대안 (Alternatives)

- **단일 앱(순수 모놀리스)**: 배치까지 한 프로세스에 넣으면 배포·자원 격리가 어렵다. 배치는
  1회 실행 후 종료하는 별도 수명주기가 필요했다.
- **진짜 마이크로서비스(저장소·배포 분리)**: 작은 팀·無 인프라 상황에 운영 부담이 과도하다.
  현재 구조는 코드 공유는 쉬우면서 프로세스만 분리한 절충안이다.

## 결과 (Consequences)

- (+) 공통 코드 재사용이 쉽고(모듈 의존), 한 번의 `gradlew build` 로 전체가 빌드된다.
- (+) 앱별로 독립 기동/배포 가능. 배치는 `CommandLineRunner` 로 떴다 종료한다.
- (−) "마이크로서비스처럼 보이지만" DB/계약이 강하게 결합되어 있다([ADR-0002](./0002-shared-h2-file-database.md), [ADR-0005](./0005-map-based-inter-service-http.md)). 진짜 분리로 가려면 추가 작업이 크다.
- 각 앱은 `scanBasePackages = "com.legacy.shop"` 로 전체 그룹을 스캔해 공통 빈을 함께 로드한다.
