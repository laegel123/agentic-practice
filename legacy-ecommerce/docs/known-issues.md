## 대형 과제 (별도 프로젝트 단위)

- ✅ **금액 `double` → `BigDecimal` 전환 — 완료(2026-06-16)** (B3 완화의 근본 해결): 6개 모듈의 금액(상품가·소계·할인·세금·합계·결제·환불·원장·장바구니 합계)을 엔티티 컬럼(`@Column(precision=19, scale=2)`)·DTO·서비스·서비스 간 HTTP·배치 누계까지 `BigDecimal`(scale 2/HALF_UP)로 전환했다. 비율(`Coupon.discountRate`)은 *금액*이 아니라 무차원 계수라 `double` 유지(곱셈 지점에서 `BigDecimal.valueOf` 무손실 변환). 회귀: 기존 characterization 단언을 `isEqualByComparingTo` 로 전환(값 보존) + `MoneyUtilsTest` scale 검증 1개 추가. ⚠️ 응답 JSON 표현 변화(`100.0`→`100.00`)·기존 파일 DB 마이그레이션은 [ADR-0006](./adr/0006-money-as-bigdecimal.md) 참고 — 멱등 `ALTER TABLE` 스크립트를 [`db/migration/`](../db/migration/) 에 커밋했다(Flyway 없음, 수동 1회 실행). [ADR-0003](./adr/0003-money-as-double.md)을 대체.
- **공유 H2 파일 DB 분리/전환**: ecommerce↔batch 공유 구조의 결합·동시성 한계. [ADR-0002](./adr/0002-shared-h2-file-database.md).
  - **1단계 — 읽기 경계 명시화 ✅(2026-06-16, [ADR-0008](./adr/0008-batch-read-only-shared-db-consumer.md))**: 물리 분리에 앞서 결합 위험을 통제. batch 리포지토리를 `ReadOnlyRepository` 로 좁혀 공유 DB 쓰기를 **타입 차원에서 차단**(save/delete/flush 제거) + 커넥션 `hikari.read-only:true` 선언 + 공유 read 계약(orders/cart/inventory 의 batch 의존 컬럼)을 `contract/shared-schema.sql` 로 문서화. 회귀: `ReadOnlyBoundaryTest`(쓰기 메서드 부재)·`SharedSchemaContractTest`(계약 컬럼 읽기, @DataJpaTest)·`ConfigExternalizationTest`(read-only 선언). 정상 동작 보존(잡들은 종전과 동일하게 읽기/집계). **물리 공유 구조는 그대로** — 기동 순서 의존·파일 DB 동시성 한계는 남는다.
  - **2단계 — 물리 분리/외부 DBMS 전환(미착수)**: 앱별 독립 DB 또는 PostgreSQL 전환 + batch 데이터 접근 재설계. 인프라 도입으로 "로컬 설정 0개" 정체성이 바뀌므로 다중 인스턴스 운영·데이터량 증가 시점에 [ADR-0002] 재검토 트리거로 다룬다. (BigDecimal 컬럼도 공유 DB 라 두 모듈을 함께 옮겨야 함, ADR-0006 참고.) ← **남은 대형 과제**
